# Beyond the Agent: Architecting a Verifiable Memory OS for Software Engineering

## Определение Целевой Архитектуры: Product Memory OS против Ложных Интерпретаций

Данная глава представляет собой фундаментальное определение целевой архитектуры, направленное на исключение любых возможностей для ее неверного понимания или упрощенной реализации. Цель — четко разграничить предлагаемую модель «Product Memory + AI Delivery Control Plane» от распространенных в индустрии аналогов, таких как планировщик на основе ИИ, оболочка для выполнения команд, база знаний или Jira-подобный инструмент. Это достигается путем явного противопоставления, анализа архитектурных антипаттернов и демонстрации того, что предложенная система является реализацией более глубокой концепции — операционной системы памяти [[58,162]]. Такая система рассматривает память не как статичное хранилище, а как управляемый системный ресурс, подобно процессорному времени или оперативной памяти, что требует соответствующего управления жизненным циклом, версионирования и безопасностью доступа [[58,59]].

Первоочередная задача при построении этой системы — сознательно избегать парадигмы, где ИИ выступает в роли автономного исполнителя, решающего задачи. Вместо этого, фокус смещается на создание среды для совместной работы человека и машины, где система служит инструментом для управления согласованностью между намерением, памятью, фактическим выполнением и реальностью проекта [[15,19]]. Этот подход кардинально отличается от многих современных агентских платформ, которые часто сфокусированы на автоматизации последовательности шагов без должного учета их долгосрочных последствий и контекста [[23,46]]. Предлагаемая архитектура позиционирует себя не как замена разработчику, а как его непартийный помощник, обеспечивающий прозрачность, контроль и возможность обучения на основе прошлого опыта.

Чтобы закрепить эту мысль, ниже представлен сравнительный анализ целевой архитектуры с наиболее распространенными ложными интерпретациями. Этот анализ имеет стратегическое значение, поскольку он служит не просто теоретической справкой, а практическим руководством для команды разработчиков, помогая ей осознанно выбирать правильный путь и избегать опасных упрощений.

| Характеристика | Product Memory OS с AI Delivery Control Plane | Типичный AI Planner | Execution Wrapper | Knowledge Base с Агентом | AI-Jira |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Центральная сущность** | Product Memory (операционное пространство проекта) [[58]] | Намерение / Задача | Командная строка / Инструмент | Знание / Факт | Задача / Карточка |
| **Основная функция** | Поддержание согласованности между намерением и реальностью | Генерация плана действий из намерения | Запуск внешних инструментов | Хранение и поиск информации | Управление статусами и потоками работ |
| **Роль агента** | Помощник, выполняющий задачи в заданном контексте, собирающий доказательства | Главный исполнитель, самостоятельно определяющий следующий шаг | Простой транспортер команд | Агент для извлечения фактов и ответов на вопросы | Автоматизация создания и перемещения задач |
| **Обработка ошибок** | Reconciliation, Review Gate, Evidence Collection, Findings [[136]] | Перезапуск/перепланирование отдельной задачи | Завершение процесса с кодом ошибки | Часто отсутствует или сводится к повторному запросу | Создание новой задачи или изменение статуса существующей |
| **Управление контекстом** | Task Context Pack, Impact Analysis, Persistent Memory Graph | Контекстное окно LLM, ограниченное размером | Прямая передача команды в shell | RAG-цепочки, ограничивающиеся retrieved documents | Этикетки, комментарии, связанные задачи |
| **Связь с реальностью** | Memory Merge, Suggested Follow-ups, Stale Risk Detection [[137]] | Изолированное выполнение, результат может не быть интегрирован | Изменение файловой системы или API, без обратной связи | Загрузка новых документов в хранилище | Создание issue, но без анализа влияния |
| **Пользовательский опыт** | Навигация по графу памяти, review найденных расхождений | Взаимодействие с интерфейсом планировщика, мониторинг прогресса | Интерфейс терминала или скрипта | Поиск по базе знаний, чат-интерфейс | Просмотр доски, изменение статусов задач |

Этот сравнительный анализ наглядно показывает, почему каждая из ложных интерпретаций является недостаточной. AI Planner решает лишь часть задачи — генерацию, но игнорирует критически важные этапы проверки и интеграции. Execution Wrapper является слишком примитивным и не обеспечивает необходимого контекста и анализа последствий. Knowledge Base с агентом не создает единого, согласованного представления о проекте, а лишь предоставляет информацию по запросу. AI-Jira, хотя и является популярным инструментом, переносит фокус с качества продукта и согласованности системы на управление рабочими процессами и метками, что не соответствует цели данного продукта.

В качестве дополнительной защиты от неверной интерпретации следует использовать список архитектурных антипаттернов — конкретных практик реализации, которых необходимо избегать любой ценой. Эти антипаттерны напрямую вытекают из требований пользователя и являются прямыми нарушениями целевой архитектуры.

**Архитектурные антипаттерны реализации:**

1.  **Задача создается напрямую из намерения без Impact Analysis.** Это прямое нарушение принципа прозорливости. Без предварительного анализа последствий система не способна предупредить о потенциально разрушительных изменениях, что делает ее небезопасной. Это антипаттерн, который превращает систему в слепого исполнителя.
2.  **Задача отправляется во внешнюю среду выполнения без Task Context Pack.** Это приводит к выполнению "вслепую", когда агент (LLM) не имеет полного понимания цели, контекста, критериев приемки или связанной памяти. Это нарушает инвариант "no execution from underspecified task".
3.  **Задача помечается как выполненная при успешном завершении процесса (`exit code 0`).** Это фундаментальный недостаток, так как успех процесса не гарантирует достижение намеченной цели. Необходим обязательный этап сбора доказательств и их проверки.
4.  **Анализ репозитория производит только сводку, а не восстановление основанной на памяти реальности.** В режиме работы с существующим проектом цель — не просто проанализировать код, а создать детальное, структурированное представление о системе в виде набора связанных объектов памяти. Это антипаттерн поверхностной аналитики.
5.  **Память обновляется неявно, без явного шага слияния.** Любые изменения в Product Memory должны происходить через контролируемый и верифицируемый процесс `Memory Merge`. Неявное обновление нарушает инвариант "memory cannot update implicitly" и делает историю изменений неясной.
6.  **Runtime adapter строится до получения одобрения отверждённого отчета о возможностях.** Реализация адаптера к внешнему инструменту "по догадке" почти гарантированно приведет к нестабильной и ненадежной работе. Верификация runtime является обязательным инженерным шагом.
7.  **Продукт спроектирован как Jira-подобный backlog tool.** Если основной рабочий процесс пользователя не сосредоточен вокруг создания и управления Change Requests и Product Memory, а скорее на перемещении задач по доске, то это является серьезным отклонением от целевой архитектуры. `Task` здесь является производным объектом, а не центральной сущностью.

Для дальнейшего углубления в философскую основу архитектуры важно рассмотреть концепцию MemOS (Memory Operating System). Исследования в этой области показывают, что переход от stateless tools к persistent agents требует наличия явного, иерархического и управляемого ресурса памяти [[58]]. MemOS предлагает рассматривать память как унифицированный, планируемый и эволюционирующий системный ресурс, аналогично тому, как традиционные ОС управляют CPU и RAM [[58,162]]. Это полностью соответствует идей, заложенных в Product Memory. Архитектура MemOS, основанная на стандартизированном абстрактном блоке `MemCube`, который служит универсальной оберткой для различных типов памяти (параметрической, активационной, текстовой), предоставляет готовую модель для формализации таких артефактов, как `Task Context Pack` [[59,142]]. `MemCube` содержит не только сам контент, но и богатую метаданные о происхождении, версии и лицензировании, что позволяет создавать надежные и прослеживаемые системы [[142]].

Таким образом, предлагаемая архитектура — это не просто набор новых компонентов, а применение продвинутых принципов проектирования операционных систем к области агентных ИИ. Она признает, что для решения сложных, долгосрочных задач требуется не просто генератор текста, а система, способная хранить, организовывать, обновлять и предоставлять доступ к своему собственному опыту. Это фундаментальное различие должно быть ясно осознаваемо всеми участниками проекта. Система должна стать "операционной системой для памяти", которая управляет информационными ресурсами проекта, обеспечивая непрерывность, согласованность и возможность обучения [[163,164]].

## Декомпозиция Жизненного Цикла: Формализация Ключевых Артефактов Выполнения

Финальная архитектурная зрелость достигается не только за счет правильных концепций, но и за счет декомпозиции процесса на явные, формализованные и проверяемые этапы. Текущая версия документа уже содержит правильные элементы, но они требуют дальнейшей проработки, чтобы устранить последние "серые зоны". Основной фокус этой главы — формализация трех критически важных артефактов, которые заменяют собой размытые понятия "планирование" и "выполнение": `Task Context Pack`, `Impact Analysis Output` и `Memory Merge Plan`. Каждый из этих артефактов должен быть описан на трех уровнях: концептуальный смысл, формальный контракт (включая примеры JSON) и явная привязка к операционным инвариантам.

**Task Context Pack: Обязательный Артефакт Выполнения**

`Task Context Pack` является самым важным из недостающих слоев, поскольку он решает проблему "слепого" выполнения, характерную для многих агентских систем. Без него реализация почти неизбежно превратится в простой pipeline вида `task -> prompt -> CLI`, что лишает систему смысла и связности. `Task Context Pack` — это первый классовый артефакт, который служит мостом между высокоуровневым намерением, зафиксированным в `Change Request`, и низкоуровневым вызовом внешней системы.

*   **Концептуальный смысл:** Его основная цель — устранить неопределенность. Он собирает воедино всю необходимую информацию для выполнения задачи, чтобы ни один агент или человек не был вынужден гадать, *почему* эта задача вообще существует. Он превращает задачу из абстрактного действия в осмысленный эксперимент с заранее определенными условиями, критериями и ожидаемыми результатами. Это напрямую поддерживает инвариант "no execution from underspecified task". Риск, который он устраняет, — это выполнение неправильной работы из-за недостаточного контекста, что является одной из самых частых причин сбоя в сложных системах [[48]].

*   **Формальный контракт:** `Task Context Pack` должен иметь строгий JSON-схему. Ниже представлен пример такой схемы.

```json

  "properties": {
    "taskIdentity": {
      "description": "Unique identifier for the task.",

      "type": "string"
    },
    "acceptanceCriteria": {

      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "id": { "type": "string" },
          "description": { "type": "string" },
          "validationMethod": { "type": "string", "enum": ["output_check", "file_exists", "regex_match", "api_response_code"] }
        },
        "required": ["id", "description"]
      }
    },

      "description": "List of memory items (capabilities, screens, etc.) that this task will affect.",
      "type": "array",
      "items": {
        "type": "object",

          "id": { "type": "string" },
          "type": { "type": "string" }

      }
    },
    "scope": {
      "description": "Explicitly defined scope of work for this task.",
      "type": "string"
    },
    "outOfScope": {
      "description": "Explicitly defined items that are out of the task's scope.",
      "type": "string"

```

Этот формальный контракт гарантирует, что каждый агент, получивший `Task Context Pack`, будет обладать полным и однозначным пониманием своей задачи. Он становится не просто инструкцией, а полноценным, версионируемым и сохраняемым артефактом, который может быть проанализирован, проверен и использован для воспроизведения выполнения.

**Impact Analysis Output: Самостоятельный Формальный Артефакт**

Второй ключевой артефакт — это результат формального анализа воздействия. Вместо того чтобы анализировать влияние реактивно, после того как задача уже начата, система должна генерировать предиктивный артефакт, который служит основой для принятия взвешенных решений.

*   **Концептуальный смысл:** `Impact Analysis Output` выводит систему из состояния реактивности в состояние прозорливости. Он анализирует `Change Request` и `Product Memory` для прогнозирования потенциальных последствий вносимых изменений. Это позволяет команде оценить "радиус поражения" изменения, выявить потенциальные конфликты зависимостей и определить, какие части системы потребуют внимания при планировании, тестировании и последующем обзоре. Это напрямую поддерживает инвариант "no serious planning before impact analysis".

*   **Формальный контракт:** Аналогично `Task Context Pack`, результат анализа воздействия должен быть формализованным JSON-объектом.

**Memory Merge Plan: Явный Шаг Обновления Памяти**

Наконец, `Memory Merge` должен быть описан как отдельный, явный пайплайн, а не как "техническая деталь" после одобрения. Это превращает его из неявного процесса в контролируемую операцию над состоянием системы.

*   **Концептуальный смысл:** После того как изменения были проверены и одобрены, `Memory Merge` — это процесс, который инкорпорирует эти изменения в `Product Memory`. Он не просто добавляет новые знания, а активно управляет состоянием всей памяти: создает новые элементы, обновляет существующие, помечает устаревшие как `stale` и корректирует их уверенность и источник происхождения. Это напрямую поддерживает инвариант "memory cannot update implicitly", делая процесс обновления видимым, контролируемым и аудируемым.

*   **Формальный контракт:** План слияния памяти также должен быть четко определен.

## Верификация и Надежность: Методологии для Runtime и Извлечения Данных

Для достижения архитектурной зрелости недостаточно просто определить концептуальные модели и формальные артефакты. Необходимо ввести строгие инженерные методологии, которые запрещают разработке идти по пути наименьшего сопротивления — "угадывания" и "примерки". Два ключевых направления, требующих такого подхода, — это взаимодействие с внешними инструментами (Runtime) и работа со старыми проектами (Existing Project Mode). Финальная ревизия должна превратить требования "провести исследование" в конкретные, повторяемые процессы с четкими выходными данными и критериями приемки.

**Методология Верификации OpenCode CLI**

Требование, чтобы адаптер к внешней среде выполнения нельзя было собирать "по догадке", является абсолютно критичным. Опыт множества агентских систем показывает, что их неспособность эффективно взаимодействовать с реальными инструментами (командной строкой, API) является одной из главных причин сбоев [[4]]. Поэтому необходимо ввести формализованную методологию верификации, которая должна стать обязательным предварительным этапом перед началом реализации любого runtime adapter.

Эта методология должна включать несколько обязательных компонентов:

1.  **Пробные шаги (Probe Steps):** Это не просто запуск нескольких команд. Необходимо систематически тестировать все аспекты поведения CLI, используя подход, аналогичный тому, что применяется в Cucumber или других BDD-фреймворках, но для CLI. Ключевые пробные шаги включают:
    *   **Стандартный ввод/вывод:** Как CLI обрабатывает передачу данных через стандартный ввод и вывод? Какие форматы (JSON, plain text, TSV) он принимает и выдает?
    *   **Завершение выполнения:** Какие коды возврата (`exit codes`) возвращаются в случае успеха, частичного успеха и ошибки? Как это повлияет на логику повторных попыток в системе?
    *   **Обработка ошибок:** Как CLI сообщает об ошибках? Это могут быть сообщения об ошибках в стандартный поток ошибок, специфические коды возврата или выход из программы. Необходимо протестировать различные сценарии ошибок (например, неверные аргументы, проблемы с сетью, недоступность ресурсов).
    *   **Аутентификация и авторизация:** Как CLI работает с OAuth, токенами, SSH-ключами? Можно ли настроить его для использования разных учетных записей? Как происходит "забывчивость" (healing loop) при истечении срока действия токена?
    *   **Лимиты скорости (Rate Limits):** Как система реагирует на превышение лимитов? Возвращает ли она специфические ошибки, которые можно обработать, или просто отказывает в обслуживании?
    *   **Логирование и сбор доказательств:** Как CLI генерирует логи? Можно ли настроить уровень детализации? Как эти логи могут быть использованы для сбора доказательств?

2.  **Создание отчета о возможностях внешней среды выполнения:** Результатом этого исследования должен быть формальный документ — `Runtime Capability Report`. Этот отчет должен быть структурирован и служить технической спецификацией для разработчиков адаптеров. Он должен точно описывать, какие команды поддерживаются, с какими параметрами, какие входные данные ожидают, какой формат выходных данных генерируют, как обрабатываются ошибки и т.д. Это своего рода "спецификация API для командной строки" [[34]].

3.  **Критерии приемки для реализации:** Реализация адаптера считается допустимой только после того, как будет получен одобрение от `Runtime Capability Report`. Это смещает фокус с "что говорит LLM" на "что делает CLI", что является фундаментальным принципом надежного инженерного проектирования [[10]]. Разработчики не могут строить свои решения на основе предположений или общих описаний; они обязаны следовать четко зафиксированному контракту.

**Полуформальная методология для режима работы с существующим проектом: контракты извлечения и консолидации**

Режим работы с существующим проектом (`existing-project mode`) является одним из сильных мест текущей архитектуры, но он требует большей формальности, чтобы избежать реализации в виде простой "умной сводки". Цель — не просто проанализировать код, а создать детальное, структурированное и основанное на памяти представление о системе. Для этого необходимо ввести полуформальные контракты извлечения и консолидации.

Эти контракты определяют, *что* извлекается, *на основе каких сигналов*, и *какой структурированный вывод* создается. Они делятся на категории, каждая из которых направлена на извлечение определенного типа артефактов.

**Extractor Categories and Rules:**

1.  **Проектная структура:**
    *   **Что извлекается:** Дерево каталогов, ключевые файлы (`package.json`, `pom.xml`, `requirements.txt`), файлы конфигурации (`.gitlab-ci.yml`, `webpack.config.js`).
    *   **На основе каких сигналов:** Структура файловой системы, содержимое манифестов пакетов, названия и расположение конфигурационных файлов.
    *   **Output Schema:** `{ "type": "ProjectStructure", "rootDirectory": "/", "directories": [...], "files": [...] }`.

2.  **Реальность бэкенда/API:**
    *   **Что извлекается:** Endpoints, методы, заголовки, тело запросов и ответов, зависимости между сервисами.
    *   **На основе каких сигналов:** Код сервера (маршруты в Express.js, Django URLs), аннотации (@GetMapping, @PostMapping), OpenAPI/Swagger спецификации.
    *   **Output Schema:** `{ "type": "APIEndpoint", "path": "/users/{id}", "method": "GET", "parameters": [...], "responses": {...} }`.

3.  **Реальность фронтенда/UI:**
    *   **Что извлекается:** Компоненты React/Vue/Angular, их пропсы и события, маршруты, CSS-классы, селекторы.
    *   **На основе каких сигналов:** Исходный код компонентов, файлы маршрутизации, CSS/SCSS файлы, DOM-структура (при наличии).
    *   **Output Schema:** `{ "type": "UIComponent", "name": "UserProfileCard", "props": [...], "events": [...] }`.

4.  **Реальность данных:**
    *   **Что извлекается:** Схемы базы данных (таблицы, поля, типы, ключи), модели ORM (Mongoose, SQLAlchemy), ETL-процессы.
    *   **На основе каких сигналов:** SQL-скрипты, определения моделей в коде, документация к данным.
    *   **Output Schema:** `{ "type": "DataEntity", "tableName": "users", "columns": [{"name": "id", "type": "int", "isPrimaryKey": true}] }`.

5.  **Реальность доставки:**
    *   **Что извлекается:** CI/CD пайплайны, шаги, условия запуска, теги коммитов.
    *   **На основе каких сигналов:** Конфигурационные файлы в `.github/workflows/`, `.circleci/config.yml`.
    *   **Output Schema:** `{ "type": "DeliveryPipelineStep", "name": "build_and_test", "trigger": "on_push_to_main" }`.

6.  **Контекстуальные декларации/документы:**
    *   **Что извлекается:** Ключевые решения (ADR), бизнес-правила, спецификации пользовательских историй, README.md.
    *   **На основе каких сигналов:** Содержимое текстовых файлов, комментарии в коде, метаданные в системах трекинга задач.
    *   **Output Schema:** `{ "type": "Decision", "title": "ADR-001: Chosen Database Technology", "content": "...", "grounding_quote": "..." }`.

**Consolidation Flow in Product Memory:**

После извлечения эти структурированные артефакты становятся основой для `Product Memory`. Процесс консолидации включает в себя:

*   **Превращение в элементы памяти:** Каждый извлеченный объект преобразуется в соответствующий элемент памяти в базе данных (Capability, Screen, API, Data Entity и т.д.). При этом сохраняется ссылка на исходный файл и строку, где был найден сигнал.
*   **Построение связей:** Между извлеченными артефактами строятся связи. Например, `UIComponent` может ссылаться на `APIEndpoint`, который, в свою очередь, ссылается на `DataEntity`. Этот процесс можно автоматизировать с помощью правил на основе совпадения имен, типов данных или специфических комментариев.
*   **Reconciliation:** На этом этапе система сравнивает извлеченное представление с тем, что известно из `Change Requests` и выполненных `Runs`. Если есть расхождения (например, в коде используется `DataEntity` "orders_v2", которого нет в извлеченной схеме), это порождает `Findings`.
*   **Превращение в корректирующие Change Requests:** `Findings` могут быть автоматически преобразованы в `Change Requests` для устранения несоответствий, например, для создания недостающей сущности в схеме данных или для исправления использования устаревшего API.

Эта полуформальная методология превращает `existing-project mode` из "магии" в повторяемый и предсказуемый инженерный процесс. Она гарантирует, что результатом будет не просто обобщение, а детальный, структурированный и взаимосвязанный граф знаний о проекте, который можно использовать для дальнейшей работы.

## Информационное Пространство: UI как операционный граф

Интерфейс пользователя (UI) в данной архитектуре не является второстепенным элементом или красивой оберткой. Он — ключевой инструмент для совместной работы человека и агента. Цель финальной ревизии — довести описание UI до уровня спецификации взаимодействия, который позволит дизайнерам и разработчикам создать не просто интерфейс, а мощное информационное пространство. Особое внимание уделяется описанию ключевых состояний экрана и событий, которые их вызывают, а также привязке к backend-контрактам. Приоритет отдается навигации по связям между сущностями, что делает `Product Memory` сильным операционным графовым пространством.

Для каждой из ключевых рабочих областей интерфейса (рабочая область, навигация по памяти, детали запроса на изменение, планирование, обзор, аудит существующего проекта) необходимо описать цель, основные блоки, ключевые действия, состояния и события.

**1. Workspace Overview (Главная страница)**

*   **Цель:** Предоставить пользователю сводный взгляд на текущее состояние всех `Change Requests`, акцентируя внимание на тех, которые требуют его внимания.
*   **Главные сущности:** `Change Request`, `Task`, `Finding`, `Run`.
*   **Основные блоки:**
    *   **Фильтры и сортировка:** По статусу (`Proposed`, `In Planning`, `Awaiting Review`, `Approved`, `Merged`, `Rejected`), по приоритету, по ответственному лицу.
    *   **Список Change Requests:** Карточки с краткой информацией: краткое описание, связанные `Tasks`, количество `Findings`, статус.
    *   **Dashboard (приборная панель):** Графики и диаграммы, показывающие общую картину (например, количество открытых CR, среднее время на этапе `Review`, распределение по типам изменений).
*   **Ключевые действия:** Создать новый `Change Request`, открыть `Change Request Detail`, перейти в `Planning & Backlog`.
*   **Важные состояния:**
    *   **Empty:** Нет `Change Requests`. Показать кнопку "Создать первый запрос на изменение".
    *   **Loading:** Индикатор загрузки списка.
    *   **Error:** Сообщение об ошибке загрузки данных.
    *   **Findings Present:** В списке CR есть элементы с нерешенными `Findings`. Иконки CR должны быть помечены соответствующим образом.
    *   **Awaiting Review:** Есть CR, ожидающие человеческого обзора. Должен быть виден фильтр "Only My Reviews".
*   **События, меняющие экран:** Создание нового `CR`, обновление статуса `CR` (через webhook или polling), добавление нового `Finding`.

**2. Product Memory Navigation (Навигация по памяти)**

*   **Цель:** Сделать `Product Memory` основным инструментом навигации и понимания системы, а не просто хранилищем. Пользователь должен легко переходить от одной сущности к другой, видя их связи.
*   **Главные сущности:** `Capability`, `Screen`, `API`, `Data Entity`, `Decision`.
*   **Основные блоки:**
    *   **Интерактивный граф:** Центральная часть экрана — веб-визуализация графа `Product Memory`. Узлы представляют сущности, ребра — связи между ними (например, "использует", "изменяет").
    *   **Панель свойств:** Когда пользователь кликает на узел или ребро, в этой панели отображаются его детали, история изменений, связанные `Evidence` и `Findings`.
    *   **Поиск:** Быстрый поиск по названиям сущностей.
*   **Ключевые действия:** Клик по узлу/ребру, поиск сущности, создание новой сущности, создание новой связи.
*   **Важные состояния:**
    *   **Graph Empty:** В памяти еще нет данных. Показать приглашение к действию ("Аудит существующего проекта для заполнения памяти").
    *   **Weak-confidence memory:** Некоторые узлы/ребра помечаются как имеющие низкую уверенность (например, извлеченные из кода, но не подтвержденные `Change Requests`). Должен быть визуальный индикатор (например, серый цвет узла).
    *   **Reconciliation produced warnings:** После выполнения `Run` система может обновить граф и пометить некоторые связи как устаревшие (`stale`) или создать новые `Findings`. Граф должен отразить эти изменения (например, серые пунктирные линии для устаревших связей).
    *   **Stale:** Узел помечен как устаревший. Должна быть кнопка "Проверить и обновить".
*   **События, меняющие экран:** Выполнение `Run`, одобрение `Review`, создание нового `Change Request`, который добавляет или изменяет сущности в графе.

**3. Change Request Detail (Детали запроса на изменение)**

*   **Цель:** Предоставить полный контекст для одного `Change Request`, включая его `Intent`, `Impact Analysis`, сгенерированные `Tasks`, а также место для `Human-in-the-loop` `Review`.
*   **Главные сущности:** `Change Request`, `ImpactAnalysisOutput`, `Task`, `TaskContextPack`, `Evidence`, `Review`.
*   **Основные блоки:**
    *   **Заголовок и описание:** Основное намерение, указанное пользователем.
    *   **Блок "Влияние":** Визуализация `Impact Analysis Output` (например, список затронутых `Capabilities` и `APIs`).
    *   **Блок "План":** Список сгенерированных `Tasks` с прогресс-баром выполнения (`Not Started`, `In Progress`, `Completed`).
    *   **Блок "Выполнение":** Для каждой `Task` отображается ее `Execution Run`: логи, вывод CLI, `Evidence` (сгенерированные файлы).
    *   **Блок "Рецензия":** Зона для отзывов. Включает `Findings` и кнопки `Approve`/`Reject`.
*   **Ключевые действия:** Запустить `Run` для `Task`, просмотреть `Evidence`, оставить комментарий, поставить `Approve`/`Reject`.
*   **События, меняющие экран:** Запуск `Run`, обновление статуса `Run`, создание нового `Evidence` или `Finding`.

**Backend Contracts (Контракты Backend):**

Для реализации этих состояний необходимо поддерживать следующие API-эндпоинты:

*   `/api/v1/change-requests`: GET (с пагинацией и фильтрами), POST (создание).
*   `/api/v1/change-requests/{cr_id}`: GET (подробная информация), PUT (обновление).
*   `/api/v1/change-requests/{cr_id}/impact-analysis`: GET (получить результат анализа).
*   `/api/v1/change-requests/{cr_id}/tasks`: GET (список задач).
*   `/api/v1/tasks/{task_id}/runs`: POST (запустить выполнение), GET (список выполнений).
*   `/api/v1/runs/{run_id}`: GET (детали выполнения, логи, evidence links).
*   `/api/v1/memory/nodes/{node_id}`: GET (получить узел для графа), PATCH (обновить узел).
*   `/api/v1/memory/edges`: POST (создать связь), DELETE (удалить связь).

Этот уровень детализации спецификации UI гарантирует, что разработчики будут создавать не просто набор экранов, а целостную экосистему взаимодействия, где каждый экран имеет свою уникальную цель и правильно реагирует на изменения в состоянии системы. Это переводит фокус с "красивого дизайна" на "функциональную эффективность", что является ключевым для продукта, ориентированного на работу с памятью.

## Управление Состоянием Проекта: слияние памяти и связь с режимом работы с существующим проектом

Эта глава объединяет два критически важных процесса, которые замыкают цикл обратной связи в системе: `Memory Merge` и `Existing-Project Mode`. `Memory Merge` является обязательным шагом жизненного цикла, который инкорпорирует изменения, одобренные в результате ревью, обратно в `Product Memory`, тем самым обогащая и актуализируя знания о проекте. `Existing-Project Mode`, в свою очередь, отвечает за первоначальное "заполнение" этой памяти, особенно в случаях, когда система внедряется в уже существующий, сложный кодовый базис. Оба процесса должны быть описаны как строгие, повторяемые и формализованные, чтобы гарантировать целостность и надежность `Product Memory` как единственной источника истины.

**Memory Merge: Обязательный Шаг Жизненного Цикла**

`Memory Merge` — это не просто техническая деталь, а фундаментальный архитектурный элемент, который обеспечивает эволюцию `Product Memory`. Его роль заключается в том, чтобы превратить временные `Evidence` и результаты `Review` в долгосрочные, структурированные знания. Этот процесс должен быть явным, контролируемым и, по возможности, атомарным.

*   **Входы:** `Memory Merge` получает на вход `Review` объект, который был успешно одобрен. На основе этого `Review` система анализирует связанные с ним `Execution Runs` и `Evidence`.

*   **Что именно делает:** Процесс `Memory Merge` состоит из нескольких шагов:
    1.  **Анализ `Evidence`:** Система анализирует сгенерированные файлы, вывод команд и другие доказательства, чтобы извлечь новые знания. Например, если задача была `Create CRUD Module`, `Evidence` может содержать новые файлы модели, контроллера и представления. Система извлекает из них информацию о новых `Data Entities`, `APIs` и `Screens`.
    2.  **Генерация плана слияния:** На основе анализа `Evidence` и связанных с `Review` `Findings` система генерирует `Memory Merge Plan`. Этот план формально описывает, какие действия нужно предпринять с `Product Memory`.
    3.  **Применение изменений:**
        *   **Создание новых элементов памяти:** Новые сущности, найденные в `Evidence`, создаются как новые элементы памяти.
        *   **Обновление существующих элементов:** Связи между существующими элементами обновляются. Например, новый `Screen` может получить ссылку на новый `API`.
        *   **Пометка как устаревших:** Если в `Evidence` был удален файл, связанный с некоторым элементом памяти, этот элемент помечается как `stale`. Также, если `Impact Analysis` предсказал риск устаревания для определенных `Memory Items`, они также помечаются.
        *   **Обновление метаданных:** Для всех измененных элементов обновляются метаданные: `freshness` (свежесть) устанавливается в `true`, `confidence` (уверенность) может быть скорректирована, а `source status` (статус источника) обновляется с `proposed` на `verified`.
    4.  **Генерация предложений по следующим шагам:** На основе `Evidence` и `Findings` система может предложить новые `Change Requests`. Например, если в результате изменений были обнаружены уязвимости безопасности, система может предложить `Change Request` для их исправления. Важно, что эти предложения генерируются, но не создаются автоматически без подтверждения человека, чтобы сохранить контроль [[119]].

*   **Возможные ошибки и конфликты:**
    *   **Конфликт версий:** Если два параллельных `Change Requests` пытаются изменить один и тот же элемент памяти, возникает конфликт. Система должна детектировать такие ситуации и помечать их как `pending_human_review`.
    *   **Несоответствие критериев приемки:** Если `Evidence`, предоставленное агентом, не соответствует `Acceptance Criteria` из `Task Context Pack`, слияние может быть отложено или отменено, и `Finding` создается для человека.
    *   **Ошибки в коде:** Если `Evidence` содержит синтаксические ошибки или нарушения стиля, это может быть зарегистрировано как `Finding` с рекомендацией исправить код.

**Связь с существующим проектом: от извлечения к консолидации**

`Existing-Project Mode` является механизмом для быстрого создания первого, пусть и неидеального, представления о проекте. Он использует методы извлечения и консолидации, описанные в предыдущей главе, чтобы "раскопать" информацию из существующих источников (код, документация, история коммитов).

*   **Как extracted artifacts превращаются в memory items:** Процесс начинается с запуска `Repo Extraction` по определенному `Extractor Contract`. Например, `API Extractor` сканирует код бэкенда и генерирует набор артефактов `APIEndpoint`. Затем `Consolidation Engine` преобразует эти артефакты в элементы памяти `API` в базе данных. Каждый элемент памяти получает `confidence` на основе того, как много подтверждающих сигналов нашлось (например, наличие аннотаций Spring, существование тестов, наличие документации в Swagger). Если сигналов мало, `confidence` остается низкой.

*   **Как между ними строятся links:** `Linking Engine` анализирует семантические связи между извлеченными артефактами. Например, он может найти, что имя переменной в коде фронтенда (`userProfileService`) совпадает с именем сервиса в бэкенде, и создать связь между `UIComponent` и `APIEndpoint`. Этот процесс может использовать комбинацию векторной семантической близости, ключевых слов и даже более сложных правил, основанных на доменной онтологии [[137]].

*   **Как reconciliation использует эти связи:** После того как `Product Memory` частично заполнена, она используется для проведения `Reconciliation`. Система выполняет `Execution Runs` (например, запускает тесты) и сравнивает фактический результат с тем, что предсказывает ее модель памяти. Если модель предсказывает, что `Screen A` должен вызывать `API B`, но в результате выполнения оказывается, что он вызывает `API C`, система регистрирует `Finding` о расхождении.

*   **Как findings превращаются в corrective Change Requests:** `Findings` — это сигналы о проблемах. Например, `Finding` может указывать на то, что в `Product Memory` есть `APIEndpoint`, который больше не существует в коде. Этот `Finding` можно автоматически преобразовать в `Change Request` типа "Удалить устаревший API из памяти". Другой `Finding` может указывать на то, что в коде используется устаревшая версия `Data Entity`. Это может инициировать `Change Request` для обновления сущности и всех, кто на нее ссылается.

Таким образом, `Memory Merge` и `Existing-Project Mode` образуют замкнутый цикл. `Existing-Project Mode` запускает процесс, создавая первоначальную память. `Memory Merge` продолжает этот процесс, постоянно обогащая и очищая память на основе реальных изменений. Это делает `Product Memory` не статичной моделью, а живым, развивающимся организмом, который адаптируется вместе с проектом.

## Границы MVP и Enforceable Invariants: Обоснование и Защита Архитектуры

Завершающая часть финальной архитектурной ревизии должна служить двум целям. Во-первых, она должна четко обосновать границы минимально жизнеспособного продукта (MVP), показывая, почему именно выбранный набор функций является необходимым и достаточным для доказательства ценности продукта. Во-вторых, она должна превратить операционные инварианты из декларативных заявлений в реально действующие механизмы, которые формируют поведение системы на всех уровнях: от жизненного цикла и сохраняемого состояния до API, выполнения и интерфейса пользователя. Это обеспечивает защиту архитектуры от "архитектурного склероза" и случайных отклонений.

**Обоснование границ MVP: Единый End-to-End Цикл**

MVP не должен быть набором разрозненных функций. Его цель — доказать одну ключевую идею: что система способна поддерживать согласованность между намерением и реальностью в рамках единого, осмысленного цикла. Все, что не помогает доказать этот цикл, является избыточной сложностью и должно быть отложено до V1.5+.

Единственный доказательный цикл, который должен быть реализован в MVP, выглядит следующим образом:

`Change Request -> Product Memory -> Impact Analysis -> Grounded Planning -> Task Context Pack -> Verified OpenCode Runtime -> Evidence -> Reconciliation -> Human Review -> Memory Merge -> Findings / Suggested Follow-ups`

Каждый шаг в этом цикле является обязательным и имеет свою ценность:

1.  **Запрос на изменение:** Центр жизненного цикла, зафиксировавший намерение.
2.  **Продуктовая память:** Основа для всего процесса, хранящая текущее состояние системы.
3.  **Анализ последствий:** Предиктивный анализ, который предотвращает непредвиденные последствия.
4.  **Основанное планирование:** Генерация задач, основанных на результате анализа.
5.  **Пакет контекста задачи:** Формальный артефакт, гарантирующий, что задача будет выполнена в правильном контексте.
6.  **Проверенная среда выполнения OpenCode:** Взаимодействие с реальным миром, которое является источником доказательств.
7.  **Доказательства:** Фактический результат выполнения, который можно проверить.
8.  **Сверка:** Сравнение фактического результата с ожидаемым, выявление расхождений.
9.  **Человеческий обзор:** Контрольный пункт, где человек подтверждает или отклоняет результат.
10. **Слияние памяти:** Интеграция верифицированного результата обратно в продуктную память.
11. **Обнаружения / Предложения по следующим шагам:** Замыкание цикла, создание новых задач на основе полученных знаний.

**Что входит в MVP (и почему):**

*   **Поддержка одного типа `Change Request` (например, "Добавить поле в сущность"):** Это позволяет сфокусироваться на одном сценарии и полностью проработать его жизненный цикл.
*   **`Existing-Project Mode` с базовыми извлекателями (структура, данные):** Необходим для создания первой версии `Product Memory`.
*   **Генерация одного `Task` на `Change Request`:** Это самый простой и понятный сценарий для начала.
*   **`Task Context Pack` с фиксированным набором полей:** Обеспечивает базовый контекст для выполнения.
*   **Простой `OpenCode Runtime` (например, для выполнения `git add`, `git commit`):** Минимально необходимый функционал для взаимодействия с системой контроля версий.
*   **`Review Gate` с кнопками `Approve`/`Reject`:** Основной контрольный пункт.
*   **Базовый `Memory Merge` с неявным обновлением (для MVP):** В MVP можно позволить простое обновление, но с полным логированием. Полностью явный `Memory Merge` лучше отложить.
*   **UI для `Workspace Overview` и `Change Request Detail`:** Позволяет пользователю видеть и управлять циклом.

**Что откладывается до V1.5+:**

*   **Сложные `Impact Analysis` (например, анализ UI/UX влияния):** Это слишком сложно для MVP. Начать стоит с технического анализа (API, данные).
*   **Автоматическая генерация нескольких `Tasks` из одного `Change Request`:** Это усложняет планирование и ревью.
*   **Сложные `Runtime Adapters` (например, для запуска тестов или деплоя):** Нужно сначала отладить базовое взаимодействие.
*   **Сложные `Memory Merge` с разрешением конфликтов:** Это сложная проблема, которую можно решать позже.
*   **Продвинутые UI-состояния (например, `weak-confidence memory`, `stale risk`):** В MVP можно использовать упрощенный граф.

**Сделать инварианты enforceable**

Существующие инварианты должны быть не просто "хорошими идеями", а механизмами, которые активно контролируют состояние системы.

*   **В жизненном цикле:** Статусы `Change Request` и `Task` должны быть жестко привязаны к этим инвариантам. Например, переход `Task` в статус `executing` возможен только тогда, когда у нее есть валидный `Task Context Pack`. Переход `Change Request` в `approved` возможен только после успешного `Review` и `Memory Merge`.

*   **В сохраняемом состоянии:** Метаданные в каждом артефакте (`Product Memory item`, `Task`, `Run`) должны включать поля для отслеживания инвариантов. Например, `memory_item.is_stale`, `task.context_pack_validated_at`.

*   **В API:** Эндпоинты должны выполнять проверки на основе инвариантов. Например, POST-запрос на создание `Task` без `Task Context Pack` должен возвращать ошибку 400 Bad Request.

*   **В выполнении:** `Runtime Adapter` должен проверять, что получает `Task Context Pack` и что он соответствует ожидаемой схеме, прежде чем выполнять команду.

*   **В ревью:** `Review Gate` должен визуально отображать нарушения инвариантов. Например, если `Evidence` не проходит проверку по критериям приемки, это должно быть ясно видно на экране ревью.

*   **В UI:** Интерфейс должен блокировать действия, которые нарушают инварианты. Например, нельзя отправить на ревью `Change Request`, если его `Impact Analysis` не завершен.

*   **В правилах использования доказательств:** `Evidence`, собранное в нарушение инвариантов (например, без `Task Context Pack`), должно помечаться как недействительное и не использоваться для `Reconciliation`.

Таким образом, финальная ревизия превращает документ из набора идей в строгую инженерную спецификацию. Она явно противопоставляет целевую архитектуру ложным интерпретациям, формализует ключевые артефакты жизненного цикла, вводит обязательные методологии верификации, детализирует UI как операционное пространство и четко определяет границы MVP. Все это делается для того, чтобы сделать документ практически неинтерпретируемым в неверном ключе, направляя команду прямо к построению сложной, надежной и действительно полезной системы — `Product Memory OS`.
